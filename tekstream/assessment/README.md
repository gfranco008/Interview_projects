# TekStream AI/ML Architect Assessment

This repository contains the full implementation for the TekStream MDR Alert Generator assessment across three tasks. Each task is self-contained but they are designed to chain: Task 1 generates the data, Task 2 trains the classifier, Task 3 builds the risk estimator on top of both.

---

## Summary of Approach

The central design constraint from the PDF is that **vendor severity is not a reliable proxy for risk**. The correlation between vendor severity and inherent severity in the generated dataset is r = 0.13 — essentially independent signals. A model that anchors on vendor scores replicates the exact bias that experienced SOC analysts learn to override. Every architectural decision flows from that constraint.

**Task 1 — Dataset design:** The synthetic generator encodes two explicit interaction effects in the label-generation logic: high inherent severity × high asset criticality increases true-positive likelihood, and high burstiness × low asset criticality increases false-positive likelihood. These are not just distributional choices — they are mechanistic hypotheses baked in so the Task 2 model can be evaluated on whether it rediscovers them. Three derived features (`contextual_risk`, `burst_index`, `severity_gap`) are pre-computed and included in the CSV to capture cross-axis signal without requiring the downstream model to learn the combinations from scratch.

**Task 2 — Model selection:** XGBoost was chosen over logistic regression for one reason: the problem has genuine nonlinear structure. Severity conflicts, burstiness thresholds, and rule-specific noise patterns do not resolve linearly even with engineered features — the 0.024 AUC gap between XGBoost (0.968) and logistic regression (0.944) quantifies that residual nonlinearity. The model is shallow (`max_depth=3`) by design: deep trees overfit the rule-identity one-hot columns and obscure the generalizable severity and context signals. Calibration is applied after model selection using `CalibratedClassifierCV` with `FrozenEstimator`, which preserves the exact fitted weights while making the output probabilities trustworthy for policy thresholds. Decision thresholds live in `policy.yaml`, not in model code, because triage rules are a governance concern and must be adjustable without retraining.

**Task 3 — Risk estimator design:** The EWMA framework was chosen over a Hidden Markov Model because the goal is operational transparency, not latent-state inference. An HMM introduces hidden states (nominal, elevated, critical) that require expectation-maximization fitting, produce non-interpretable state assignments, and resist straightforward explanation to analysts. The EWMA update equation is decomposable at every step: each risk score is a direct function of inherent severity, asset criticality, behavioral context, and disposition, weighted explicitly. Inherent severity carries 0.45 of the total contribution weight; vendor severity carries 0.10. This encoding ensures the risk estimator cannot be inflated by vendor noise. The kill-chain progression bonus — counting ordered MITRE stage pairs within a 24-hour window — adds sensitivity to coordinated multi-stage campaigns without requiring a full sequence model.

## Quick Start

### Task 1 — Generate the synthetic dataset

```
python generate_dataset.py --rows 10000 --seed 42
```

Optional:
- `--extra-columns` adds SOC-style context columns
- `--config path/to/config.json` overrides defaults

Validate the dataset:

```
python validate_dataset.py --rows 2000
```

Add `--strict` to exit non-zero if any validation warnings are found.

### Task 2 — Train the false-positive predictor

```
python fp_predictor.py
```

Trains a calibrated XGBoost classifier (compared against logistic regression), applies externalized decision thresholds from `policy.yaml`, and produces all evaluation plots and reports.

### Task 3 — Run the risk estimator

```
python risk_estimator.py
```

Runs the EWMA risk state estimator with severity weighting and kill-chain progression bonus on the generated dataset.

## Interaction Effects Implemented

Two explicit interaction effects are baked into the label generation logic in Task 1. The Task 2 model is then evaluated on whether it rediscovers them via SHAP.

**Effect 1 — Severity × Criticality → True Positive**
High inherent severity combined with high asset criticality is a compounding risk signal. The `contextual_risk` derived feature captures this as a product interaction.
- High/high subset TP rate: **58.5%** vs baseline 15.0%

**Effect 2 — Burstiness × Low Criticality → False Positive**
Bursty alert storms on low-criticality assets are characteristic of noisy rules. The `low_crit_burst` derived feature captures this.
- High burst / low criticality FP rate: **96.9%** vs baseline 75.0%

## Derived Features

| Feature | Formula | Direction |
| --- | --- | --- |
| `contextual_risk` | `inherent_severity × asset_criticality` | Higher → fewer false positives |
| `burst_index` | `log1p(alert_count_1h) / log1p(alert_count_24h)` | Higher → more false positives |
| `severity_gap` | `inherent_severity − vendor_severity` | Negative gap → likely false positive |
